# Semantic HTML and Accessibility

## Learning Objectives
- Understand the relationship between semantic HTML and accessibility
- Master the accessibility tree and how browsers expose semantics to assistive technologies
- Learn implicit ARIA roles provided by HTML elements
- Apply semantic HTML patterns to improve accessibility
- Recognize when semantic HTML is sufficient vs when ARIA is needed

## Introduction to Semantic HTML

Semantic HTML uses elements that clearly describe their meaning and purpose both to browsers and developers. Unlike generic `<div>` and `<span>` elements, semantic elements like `<nav>`, `<article>`, and `<button>` convey inherent meaning about their content and role in the document structure.

The relationship between semantic HTML and accessibility is fundamental: semantic elements provide built-in accessibility features that would otherwise require manual ARIA attributes and JavaScript. When you use a `<button>` element, you automatically get keyboard support, focus management, and proper screen reader announcements. When you use a `<div>` styled to look like a button, you get none of these features.

The accessibility tree is the bridge between your HTML and assistive technologies. Browsers parse your HTML and create two parallel structures: the DOM (Document Object Model) for visual rendering and the accessibility tree for assistive technologies. Semantic HTML ensures this accessibility tree is rich, meaningful, and accurately represents your interface structure.

## The Accessibility Tree

The accessibility tree is a data structure generated by the browser that represents the semantic structure and properties of a page. It's derived from the DOM but contains only accessibility-relevant information.

```
Visual DOM Tree              Accessibility Tree
┌─────────────┐              ┌─────────────────────┐
│   <html>    │              │   document          │
│     │       │              │      │              │
│   <body>    │              │   region (main)     │
│     │       │              │      │              │
│   <main>────┼─────────────>│   ├─ heading (h1)   │
│     │       │              │   │  "Welcome"      │
│   <h1>      │              │   │                 │
│   <nav>─────┼─────────────>│   └─ navigation     │
│     │       │              │       ├─ link       │
│   <ul>      │              │       ├─ link       │
│   <li>      │              │       └─ link       │
│   <a>       │              │                     │
│   <div>     │ (skipped)    │                     │
│   <span>    │ (skipped)    │                     │
└─────────────┘              └─────────────────────┘
```

### How the Accessibility Tree Works

```typescript
// React: Example showing DOM vs Accessibility Tree
const PageStructure: React.FC = () => {
  return (
    <div> {/* Generic container - minimal a11y info */}
      <header> {/* Landmark: banner role */}
        <h1>Site Title</h1> {/* Heading level 1 */}
        <nav> {/* Landmark: navigation role */}
          <ul> {/* List */}
            <li><a href="/">Home</a></li> {/* Link */}
            <li><a href="/about">About</a></li>
          </ul>
        </nav>
      </header>
      
      <main> {/* Landmark: main role */}
        <article> {/* Article region */}
          <h2>Article Title</h2> {/* Heading level 2 */}
          <p>Content...</p> {/* Text */}
        </article>
      </main>
      
      <footer> {/* Landmark: contentinfo role */}
        <p>&copy; 2024</p>
      </footer>
    </div>
  );
};
```

## Implicit ARIA Roles

Every semantic HTML element has an implicit ARIA role. Understanding these roles helps you choose the right element and know when you need explicit ARIA.

```typescript
// React: Semantic elements and their implicit roles
const SemanticRolesExample: React.FC = () => {
  return (
    <>
      {/* Landmark Roles */}
      <header> {/* role="banner" (when top-level) */}
        <h1>Page Title</h1>
      </header>
      
      <nav> {/* role="navigation" */}
        <ul>
          <li><a href="/">Home</a></li>
        </ul>
      </nav>
      
      <main> {/* role="main" */}
        {/* Document Structure Roles */}
        <article> {/* role="article" */}
          <h2>Article Title</h2> {/* role="heading" aria-level="2" */}
          
          <section> {/* role="region" (when labeled) */}
            <h3>Section Title</h3>
          </section>
          
          <aside> {/* role="complementary" */}
            <p>Related content</p>
          </aside>
        </article>
        
        {/* List Roles */}
        <ul> {/* role="list" */}
          <li>Item 1</li> {/* role="listitem" */}
          <li>Item 2</li>
        </ul>
        
        <ol> {/* role="list" */}
          <li>First</li> {/* role="listitem" */}
        </ol>
        
        <dl> {/* role="group" */}
          <dt>Term</dt> {/* role="term" */}
          <dd>Definition</dd> {/* role="definition" */}
        </dl>
        
        {/* Interactive Elements */}
        <button>Click</button> {/* role="button" */}
        <a href="/page">Link</a> {/* role="link" */}
        
        {/* Form Elements */}
        <form> {/* role="form" (when labeled) */}
          <label htmlFor="name">Name</label>
          <input 
            type="text" 
            id="name" 
            // role="textbox"
          />
          
          <input 
            type="checkbox" 
            id="agree"
            // role="checkbox"
          />
          
          <select> {/* role="combobox" or "listbox" */}
            <option>Option 1</option> {/* role="option" */}
          </select>
          
          <textarea></textarea> {/* role="textbox" multiline="true" */}
          
          <input type="submit" /> {/* role="button" */}
        </form>
        
        {/* Table Elements */}
        <table> {/* role="table" */}
          <thead> {/* role="rowgroup" */}
            <tr> {/* role="row" */}
              <th>Header</th> {/* role="columnheader" */}
            </tr>
          </thead>
          <tbody> {/* role="rowgroup" */}
            <tr> {/* role="row" */}
              <td>Data</td> {/* role="cell" */}
            </tr>
          </tbody>
        </table>
        
        {/* Text Content */}
        <p>Paragraph</p> {/* role="paragraph" */}
        <strong>Bold</strong> {/* role="strong" */}
        <em>Italic</em> {/* role="emphasis" */}
        <code>Code</code> {/* role="code" */}
        <img src="photo.jpg" alt="Description" /> {/* role="img" */}
        
        {/* Dialog and Modal */}
        <dialog open> {/* role="dialog" */}
          <p>Dialog content</p>
        </dialog>
      </main>
      
      <footer> {/* role="contentinfo" (when top-level) */}
        <p>&copy; 2024</p>
      </footer>
    </>
  );
};
```

## Semantic HTML Patterns

### Pattern 1: Proper Heading Hierarchy

Headings create an outline of your document that screen reader users navigate.

```typescript
// React: Proper heading hierarchy
const ArticlePage: React.FC = () => {
  return (
    <main>
      <article>
        {/* Level 1: Page title */}
        <h1>Understanding React Hooks</h1>
        
        {/* Level 2: Major sections */}
        <section>
          <h2>What are Hooks?</h2>
          <p>Hooks are functions that...</p>
        </section>
        
        <section>
          <h2>Common Hooks</h2>
          
          {/* Level 3: Subsections */}
          <section>
            <h3>useState</h3>
            <p>The useState hook...</p>
            
            {/* Level 4: Sub-subsections */}
            <h4>Basic Example</h4>
            <pre><code>const [count, setCount] = useState(0);</code></pre>
            
            <h4>Advanced Usage</h4>
            <pre><code>const [state, setState] = useState(() => expensiveCalculation());</code></pre>
          </section>
          
          <section>
            <h3>useEffect</h3>
            <p>The useEffect hook...</p>
          </section>
        </section>
        
        <section>
          <h2>Best Practices</h2>
          <ul>
            <li>Don't skip heading levels</li>
            <li>Use only one h1 per page</li>
          </ul>
        </section>
      </article>
    </main>
  );
};

// ❌ BAD: Skipping heading levels
const BadHeadingExample: React.FC = () => {
  return (
    <article>
      <h1>Title</h1>
      <h3>Skipped h2!</h3> {/* Don't skip from h1 to h3 */}
      <h2>Out of order</h2> {/* h2 should come before h3 */}
    </article>
  );
};

// ✅ GOOD: Proper hierarchy
const GoodHeadingExample: React.FC = () => {
  return (
    <article>
      <h1>Title</h1>
      <h2>Section</h2>
      <h3>Subsection</h3>
      <h2>Another Section</h2>
    </article>
  );
};
```

### Pattern 2: Landmarks and Page Regions

Landmarks allow screen reader users to quickly navigate to different sections of a page.

```typescript
// React: Comprehensive landmark structure
const LandmarkStructure: React.FC = () => {
  return (
    <>
      {/* Banner landmark (header) - only one per page */}
      <header>
        <div className="logo">
          <img src="/logo.svg" alt="Company Name" />
        </div>
        
        {/* Navigation landmark - can have multiple with labels */}
        <nav aria-label="Primary navigation">
          <ul>
            <li><a href="/">Home</a></li>
            <li><a href="/products">Products</a></li>
            <li><a href="/about">About</a></li>
          </ul>
        </nav>
      </header>
      
      {/* Main landmark - only one per page */}
      <main>
        {/* Search landmark */}
        <div role="search">
          <form>
            <label htmlFor="site-search">Search site</label>
            <input type="search" id="site-search" />
            <button type="submit">Search</button>
          </form>
        </div>
        
        <article>
          <h1>Article Title</h1>
          <p>Main content...</p>
          
          {/* Complementary landmark (aside) */}
          <aside aria-labelledby="sidebar-heading">
            <h2 id="sidebar-heading">Related Articles</h2>
            <ul>
              <li><a href="/article-1">Article 1</a></li>
              <li><a href="/article-2">Article 2</a></li>
            </ul>
          </aside>
        </article>
      </main>
      
      {/* Contentinfo landmark (footer) - only one per page */}
      <footer>
        <nav aria-label="Footer navigation">
          <ul>
            <li><a href="/privacy">Privacy</a></li>
            <li><a href="/terms">Terms</a></li>
          </ul>
        </nav>
        <p>&copy; 2024 Company Name</p>
      </footer>
    </>
  );
};
```

```
Landmark Navigation Structure:
┌────────────────────────────────────┐
│ header (banner landmark)           │
│   ├─ logo                          │
│   └─ nav (navigation landmark)     │
├────────────────────────────────────┤
│ main (main landmark)               │
│   ├─ search (search landmark)      │
│   └─ article                       │
│       └─ aside (complementary)     │
├────────────────────────────────────┤
│ footer (contentinfo landmark)      │
│   └─ nav (navigation landmark)     │
└────────────────────────────────────┘

Screen reader shortcut keys:
- h: Next heading
- 1-6: Next heading of specific level
- l: Next list
- i: Next list item
- b: Next button
- f: Next form field
- d: Next landmark
```

### Pattern 3: Lists and List Items

Lists provide structure and help screen readers announce the number of items.

```typescript
// React: Proper list usage
const ListExamples: React.FC = () => {
  const products = [
    { id: 1, name: 'Laptop', price: 999 },
    { id: 2, name: 'Mouse', price: 29 },
    { id: 3, name: 'Keyboard', price: 79 }
  ];
  
  const instructions = [
    'Preheat oven to 350°F',
    'Mix dry ingredients',
    'Add wet ingredients',
    'Bake for 25 minutes'
  ];
  
  return (
    <div>
      {/* Unordered list for non-sequential items */}
      <ul>
        {products.map(product => (
          <li key={product.id}>
            <h3>{product.name}</h3>
            <p>${product.price}</p>
          </li>
        ))}
      </ul>
      
      {/* Ordered list for sequential steps */}
      <ol>
        {instructions.map((step, index) => (
          <li key={index}>{step}</li>
        ))}
      </ol>
      
      {/* Description list for term-definition pairs */}
      <dl>
        <dt>HTML</dt>
        <dd>HyperText Markup Language</dd>
        
        <dt>CSS</dt>
        <dd>Cascading Style Sheets</dd>
        
        <dt>JS</dt>
        <dd>JavaScript</dd>
      </dl>
    </div>
  );
};

// ❌ BAD: Using divs instead of lists
const BadListExample: React.FC = () => {
  return (
    <div>
      <div>Item 1</div>
      <div>Item 2</div>
      <div>Item 3</div>
      {/* Screen readers won't announce this as a list */}
    </div>
  );
};

// ✅ GOOD: Proper list markup
const GoodListExample: React.FC = () => {
  return (
    <ul>
      <li>Item 1</li>
      <li>Item 2</li>
      <li>Item 3</li>
      {/* Screen reader: "List, 3 items" */}
    </ul>
  );
};
```

### Pattern 4: Tables for Tabular Data

Tables should be used for actual tabular data, not layout. Proper table markup provides context for each cell.

```typescript
// React: Accessible data table
interface User {
  id: number;
  name: string;
  email: string;
  role: string;
  status: 'active' | 'inactive';
}

const UsersTable: React.FC<{ users: User[] }> = ({ users }) => {
  return (
    <table>
      {/* Caption provides context for the table */}
      <caption>User Directory (Total: {users.length} users)</caption>
      
      <thead>
        <tr>
          {/* th elements define column headers */}
          <th scope="col">Name</th>
          <th scope="col">Email</th>
          <th scope="col">Role</th>
          <th scope="col">Status</th>
          <th scope="col">Actions</th>
        </tr>
      </thead>
      
      <tbody>
        {users.map(user => (
          <tr key={user.id}>
            {/* First cell as row header */}
            <th scope="row">{user.name}</th>
            <td>{user.email}</td>
            <td>{user.role}</td>
            <td>
              <span className={`status-${user.status}`}>
                {user.status}
              </span>
            </td>
            <td>
              <button 
                aria-label={`Edit ${user.name}`}
                onClick={() => console.log(`Edit ${user.id}`)}
              >
                Edit
              </button>
              <button 
                aria-label={`Delete ${user.name}`}
                onClick={() => console.log(`Delete ${user.id}`)}
              >
                Delete
              </button>
            </td>
          </tr>
        ))}
      </tbody>
      
      {/* Optional footer for summary data */}
      <tfoot>
        <tr>
          <td colSpan={5}>
            Showing {users.length} of {users.length} users
          </td>
        </tr>
      </tfoot>
    </table>
  );
};
```

### Pattern 5: Forms and Form Controls

Semantic form elements provide built-in accessibility and validation.

```typescript
// React: Accessible form with semantic elements
const RegistrationForm: React.FC = () => {
  const [formData, setFormData] = useState({
    username: '',
    email: '',
    password: '',
    country: '',
    newsletter: false,
    role: 'user'
  });

  return (
    <form onSubmit={(e) => e.preventDefault()}>
      <fieldset>
        <legend>Account Information</legend>
        
        {/* Text input */}
        <div>
          <label htmlFor="username">
            Username
            <abbr title="required" aria-label="required">*</abbr>
          </label>
          <input
            type="text"
            id="username"
            name="username"
            value={formData.username}
            onChange={(e) => setFormData({...formData, username: e.target.value})}
            required
            aria-required="true"
          />
        </div>
        
        {/* Email input */}
        <div>
          <label htmlFor="email">
            Email Address
            <abbr title="required" aria-label="required">*</abbr>
          </label>
          <input
            type="email"
            id="email"
            name="email"
            value={formData.email}
            onChange={(e) => setFormData({...formData, email: e.target.value})}
            required
            aria-required="true"
            autoComplete="email"
          />
        </div>
        
        {/* Password input */}
        <div>
          <label htmlFor="password">
            Password
            <abbr title="required" aria-label="required">*</abbr>
          </label>
          <input
            type="password"
            id="password"
            name="password"
            value={formData.password}
            onChange={(e) => setFormData({...formData, password: e.target.value})}
            required
            aria-required="true"
            autoComplete="new-password"
            minLength={8}
          />
          <small id="password-hint">
            Must be at least 8 characters
          </small>
        </div>
      </fieldset>
      
      <fieldset>
        <legend>Preferences</legend>
        
        {/* Select dropdown */}
        <div>
          <label htmlFor="country">Country</label>
          <select
            id="country"
            name="country"
            value={formData.country}
            onChange={(e) => setFormData({...formData, country: e.target.value})}
          >
            <option value="">Select a country</option>
            <option value="us">United States</option>
            <option value="uk">United Kingdom</option>
            <option value="ca">Canada</option>
          </select>
        </div>
        
        {/* Checkbox */}
        <div>
          <label>
            <input
              type="checkbox"
              name="newsletter"
              checked={formData.newsletter}
              onChange={(e) => setFormData({...formData, newsletter: e.target.checked})}
            />
            Subscribe to newsletter
          </label>
        </div>
        
        {/* Radio buttons */}
        <fieldset>
          <legend>Account Type</legend>
          <div>
            <label>
              <input
                type="radio"
                name="role"
                value="user"
                checked={formData.role === 'user'}
                onChange={(e) => setFormData({...formData, role: e.target.value})}
              />
              Regular User
            </label>
          </div>
          <div>
            <label>
              <input
                type="radio"
                name="role"
                value="admin"
                checked={formData.role === 'admin'}
                onChange={(e) => setFormData({...formData, role: e.target.value})}
              />
              Administrator
            </label>
          </div>
        </fieldset>
      </fieldset>
      
      <div>
        <button type="submit">Create Account</button>
        <button type="reset">Reset Form</button>
      </div>
    </form>
  );
};
```

### Pattern 6: Buttons vs Links

Buttons and links have different semantic meanings and should be used appropriately.

```typescript
// React: Proper button and link usage
const ButtonVsLink: React.FC = () => {
  const handleDelete = () => {
    console.log('Deleting item...');
  };

  return (
    <div>
      {/* ✅ Links: Navigation to another page or section */}
      <a href="/products">
        View All Products
      </a>
      
      <a href="/product/123">
        View Product Details
      </a>
      
      <a href="#section-2">
        Jump to Section 2
      </a>
      
      {/* ✅ Buttons: Actions and interactions */}
      <button type="button" onClick={handleDelete}>
        Delete Item
      </button>
      
      <button type="submit">
        Submit Form
      </button>
      
      <button type="button" onClick={() => console.log('Toggle')}>
        Toggle Menu
      </button>
      
      {/* ❌ BAD: Link styled as button for an action */}
      <a href="#" onClick={(e) => {
        e.preventDefault();
        handleDelete();
      }}>
        Delete (Don't do this!)
      </a>
      
      {/* ❌ BAD: Button for navigation */}
      <button onClick={() => window.location.href = '/products'}>
        View Products (Use a link instead!)
      </button>
      
      {/* ✅ GOOD: If you must use a button for navigation, use proper semantics */}
      <button 
        type="button"
        onClick={() => window.location.href = '/products'}
        role="link"
      >
        View Products
      </button>
    </div>
  );
};
```

```
Decision Tree: Button vs Link
┌─────────────────────────────────┐
│ Does it navigate to a new page  │
│ or location?                    │
└────────┬─────────────────┬──────┘
         │ YES             │ NO
         ▼                 ▼
    ┌────────┐       ┌──────────┐
    │  Link  │       │  Button  │
    │  <a>   │       │ <button> │
    └────────┘       └──────────┘
    
Examples:
Links:                Buttons:
- View details        - Submit form
- Read more          - Delete item
- Go to page         - Toggle menu
- Download file      - Open modal
- Jump to section    - Add to cart
```

## When Semantic HTML Is Not Enough

Sometimes you need custom interactive components that don't have native HTML equivalents. In these cases, ARIA is necessary.

```typescript
// React: Custom accordion requiring ARIA
const Accordion: React.FC = () => {
  const [openPanels, setOpenPanels] = useState<Set<number>>(new Set());
  
  const togglePanel = (index: number) => {
    const newPanels = new Set(openPanels);
    if (newPanels.has(index)) {
      newPanels.delete(index);
    } else {
      newPanels.add(index);
    }
    setOpenPanels(newPanels);
  };
  
  const panels = [
    { title: 'What is React?', content: 'React is a JavaScript library...' },
    { title: 'What are Hooks?', content: 'Hooks are functions that...' },
    { title: 'What is JSX?', content: 'JSX is a syntax extension...' }
  ];
  
  return (
    <div className="accordion">
      {panels.map((panel, index) => {
        const isOpen = openPanels.has(index);
        const headingId = `accordion-heading-${index}`;
        const panelId = `accordion-panel-${index}`;
        
        return (
          <div key={index} className="accordion-item">
            <h3 id={headingId}>
              <button
                type="button"
                aria-expanded={isOpen}
                aria-controls={panelId}
                onClick={() => togglePanel(index)}
                className="accordion-trigger"
              >
                {panel.title}
                <span aria-hidden="true">{isOpen ? '−' : '+'}</span>
              </button>
            </h3>
            
            <div
              id={panelId}
              role="region"
              aria-labelledby={headingId}
              hidden={!isOpen}
              className="accordion-panel"
            >
              <p>{panel.content}</p>
            </div>
          </div>
        );
      })}
    </div>
  );
};

// Angular: Custom tabs requiring ARIA
@Component({
  selector: 'app-tabs',
  template: `
    <div class="tabs">
      <div role="tablist" aria-label="Content sections">
        <button
          *ngFor="let tab of tabs; let i = index"
          role="tab"
          [attr.id]="'tab-' + i"
          [attr.aria-selected]="activeTab === i"
          [attr.aria-controls]="'panel-' + i"
          [attr.tabindex]="activeTab === i ? 0 : -1"
          (click)="selectTab(i)"
          (keydown)="handleKeyDown($event, i)"
          class="tab"
        >
          {{ tab.label }}
        </button>
      </div>
      
      <div
        *ngFor="let tab of tabs; let i = index"
        role="tabpanel"
        [attr.id]="'panel-' + i"
        [attr.aria-labelledby]="'tab-' + i"
        [hidden]="activeTab !== i"
        [attr.tabindex]="0"
        class="tab-panel"
      >
        <ng-container *ngIf="activeTab === i">
          {{ tab.content }}
        </ng-container>
      </div>
    </div>
  `,
  styles: [`
    .tabs {
      border: 1px solid #ccc;
    }
    
    [role="tablist"] {
      display: flex;
      border-bottom: 1px solid #ccc;
    }
    
    .tab {
      padding: 0.5rem 1rem;
      border: none;
      background: none;
      cursor: pointer;
    }
    
    .tab[aria-selected="true"] {
      background: white;
      border-bottom: 3px solid #005fcc;
    }
    
    .tab:focus {
      outline: 2px solid #005fcc;
      outline-offset: -2px;
    }
    
    .tab-panel {
      padding: 1rem;
    }
  `]
})
export class TabsComponent {
  @Input() tabs: Array<{ label: string; content: string }> = [];
  activeTab = 0;
  
  selectTab(index: number): void {
    this.activeTab = index;
  }
  
  handleKeyDown(event: KeyboardEvent, currentIndex: number): void {
    let newIndex = currentIndex;
    
    switch (event.key) {
      case 'ArrowRight':
        newIndex = (currentIndex + 1) % this.tabs.length;
        break;
      case 'ArrowLeft':
        newIndex = currentIndex === 0 ? this.tabs.length - 1 : currentIndex - 1;
        break;
      case 'Home':
        newIndex = 0;
        break;
      case 'End':
        newIndex = this.tabs.length - 1;
        break;
      default:
        return;
    }
    
    event.preventDefault();
    this.selectTab(newIndex);
    
    // Focus the newly selected tab
    setTimeout(() => {
      const tab = document.getElementById(`tab-${newIndex}`);
      tab?.focus();
    }, 0);
  }
}
```

## Common Mistakes

### Mistake 1: Using Divs for Everything

```typescript
// ❌ BAD: Div soup with no semantics
const BadExample: React.FC = () => {
  return (
    <div className="header">
      <div className="nav">
        <div className="nav-item">Home</div>
        <div className="nav-item">About</div>
      </div>
      <div className="main">
        <div className="article">
          <div className="title">Article Title</div>
          <div className="content">Content...</div>
        </div>
      </div>
      <div className="footer">
        Copyright 2024
      </div>
    </div>
  );
};

// ✅ GOOD: Semantic elements
const GoodExample: React.FC = () => {
  return (
    <>
      <header>
        <nav>
          <a href="/">Home</a>
          <a href="/about">About</a>
        </nav>
      </header>
      <main>
        <article>
          <h1>Article Title</h1>
          <p>Content...</p>
        </article>
      </main>
      <footer>
        <p>Copyright 2024</p>
      </footer>
    </>
  );
};
```

### Mistake 2: Missing Form Labels

```typescript
// ❌ BAD: No label association
<input type="text" placeholder="Enter your name" />

// ✅ GOOD: Properly labeled
<label htmlFor="name-input">Name</label>
<input id="name-input" type="text" placeholder="Enter your name" />

// ✅ ALSO GOOD: Implicit label
<label>
  Name
  <input type="text" placeholder="Enter your name" />
</label>
```

### Mistake 3: Adding Unnecessary ARIA

```typescript
// ❌ BAD: Redundant ARIA
<nav role="navigation">
  <ul role="list">
    <li role="listitem">
      <a href="/" role="link">Home</a>
    </li>
  </ul>
</nav>

// ✅ GOOD: Semantic HTML provides implicit roles
<nav>
  <ul>
    <li>
      <a href="/">Home</a>
    </li>
  </ul>
</nav>
```

### Mistake 4: Incorrect Heading Hierarchy

```typescript
// ❌ BAD: Skipping levels
<h1>Page Title</h1>
<h3>Skipped h2!</h3>
<h4>Subsection</h4>

// ✅ GOOD: Proper hierarchy
<h1>Page Title</h1>
<h2>Section</h2>
<h3>Subsection</h3>
```

### Mistake 5: Using Tables for Layout

```typescript
// ❌ BAD: Table for layout
<table>
  <tr>
    <td>Sidebar</td>
    <td>Main content</td>
  </tr>
</table>

// ✅ GOOD: CSS layout
<div className="layout">
  <aside>Sidebar</aside>
  <main>Main content</main>
</div>
```

## Best Practices

1. **Start with semantic HTML**: Always choose the most appropriate semantic element
2. **Use landmarks**: Provide clear page structure with header, nav, main, aside, footer
3. **Maintain heading hierarchy**: Don't skip heading levels
4. **Label all form inputs**: Use explicit or implicit label associations
5. **Use buttons for actions, links for navigation**: Follow semantic meanings
6. **Provide context with lists**: Use ul, ol, and dl appropriately
7. **Tables for tabular data only**: Include caption, th with scope
8. **Test the accessibility tree**: Use browser DevTools to inspect
9. **Only add ARIA when necessary**: First rule of ARIA is "don't use ARIA"
10. **Validate HTML**: Use W3C validator to catch structural errors

## When to Use Semantic HTML

Use semantic HTML:
- For all standard page structures and content
- When native elements provide the functionality you need
- For SEO benefits and better discoverability
- To reduce the amount of ARIA you need to write
- To get built-in keyboard support and focus management

## When Semantic HTML Is Not Enough

You may need ARIA when:
- Building custom widgets (accordions, tabs, tree views)
- Creating dynamic content that changes without page reload
- Providing additional context not conveyed by HTML alone
- Implementing patterns not available as native elements
- Working with legacy code that can't be restructured

## Interview Questions

**Q1: What is the accessibility tree and how does it differ from the DOM?**

**A:** The accessibility tree is a data structure generated by browsers that represents the semantic structure and accessibility properties of a page. While the DOM (Document Object Model) represents all HTML elements and is used for visual rendering, the accessibility tree is derived from the DOM but contains only accessibility-relevant information.

The accessibility tree includes:
- Semantic roles (button, link, heading, etc.)
- Accessible names (labels, alt text)
- States and properties (checked, expanded, disabled)
- Relationships between elements

Generic elements like `<div>` and `<span>` often don't appear in the accessibility tree unless they have explicit ARIA roles or are focusable. Semantic elements like `<button>`, `<nav>`, and `<h1>` automatically populate the accessibility tree with meaningful information.

Screen readers and other assistive technologies consume the accessibility tree rather than directly parsing the DOM, which is why semantic HTML is crucial for accessibility.

**Q2: What are implicit ARIA roles and why are they important?**

**A:** Implicit ARIA roles are the accessibility roles automatically assigned to semantic HTML elements without requiring explicit role attributes. For example:
- `<button>` has implicit role="button"
- `<nav>` has implicit role="navigation"
- `<main>` has implicit role="main"
- `<article>` has implicit role="article"

These implicit roles are important because:

1. **Reduce code complexity**: You don't need to manually add role attributes
2. **Better browser support**: Native elements have better cross-browser compatibility
3. **Built-in behavior**: Semantic elements include keyboard support and other behaviors
4. **Less error-prone**: You can't accidentally misconfigure implicit roles
5. **Forward compatibility**: As accessibility standards evolve, browsers update native element behavior

The first rule of ARIA is "Don't use ARIA if a native HTML element provides the semantics you need." Implicit roles are a key reason for this rule.

**Q3: How should you structure heading levels in a web page?**

**A:** Heading structure should follow a logical hierarchy:

1. **One h1 per page**: Represents the main page title or primary content heading
2. **Don't skip levels**: Go from h1 to h2 to h3, never skip (h1 to h3)
3. **Multiple same-level headings are fine**: You can have multiple h2s under an h1
4. **Can go back up levels**: After an h4, you can go back to h2 for a new section
5. **Represent content structure**: Headings create an outline of your content

Example of proper structure:
```
h1: Complete Guide to React
  h2: Getting Started
    h3: Installation
    h3: Your First Component
  h2: Core Concepts
    h3: Components
      h4: Function Components
      h4: Class Components
    h3: Props
    h3: State
  h2: Advanced Topics
```

Screen reader users navigate by headings using shortcut keys (H for next heading, 1-6 for specific levels), so proper hierarchy is essential for navigation and understanding content structure.

**Q4: When should you use a button vs a link?**

**A:** The choice between button and link depends on the semantic meaning of the action:

**Use `<a>` (link) when:**
- Navigating to a different page or URL
- Jumping to a section on the same page (anchors)
- Downloading a file
- Opening an email client (mailto:)
- Making a phone call (tel:)
- The action has a distinct URL that can be bookmarked or shared

**Use `<button>` when:**
- Performing an action on the current page (submit, delete, toggle)
- Opening a modal or dialog
- Expanding/collapsing content
- Triggering JavaScript functionality
- Submitting a form
- The action doesn't have a navigable URL

**Why this matters:**
- Links support "Open in new tab", "Copy link", "Bookmark"
- Buttons are announced differently by screen readers
- Default keyboard behavior differs (Enter for both, Space only for buttons)
- Browsers handle them differently (middle-click, right-click menus)
- Search engines treat them differently for crawling

**Common mistake:** Using `<a href="#" onClick={handler}>` for actions. Use a button instead:
```typescript
// ❌ BAD
<a href="#" onClick={handleDelete}>Delete</a>

// ✅ GOOD
<button onClick={handleDelete}>Delete</button>
```

**Q5: What are landmark regions and why are they important for accessibility?**

**A:** Landmark regions are HTML elements that define major sections of a page, allowing screen reader users to quickly navigate to different areas. The main landmarks are:

- **banner** (`<header>` at top level): Site header with logo, site title
- **navigation** (`<nav>`): Navigation links
- **main** (`<main>`): Primary content (only one per page)
- **complementary** (`<aside>`): Related but separate content
- **contentinfo** (`<footer>` at top level): Site footer with copyright, links
- **search** (`<form role="search">`): Search functionality
- **region** (`<section>` with label): Generic landmark for important areas

**Why they're important:**

1. **Quick navigation**: Screen readers provide landmark navigation shortcuts (D key in many screen readers)
2. **Page overview**: Users can hear a list of all landmarks to understand page structure
3. **Skip repetitive content**: Jump directly to main content, bypassing navigation
4. **Better orientation**: Users know what part of the page they're in

**Best practices:**
- Use only one `<main>` per page
- Use only one top-level `<header>` and `<footer>` (banner and contentinfo)
- Label multiple landmarks of the same type with `aria-label`
- Don't overuse - only for truly significant page sections

Example:
```typescript
<header>  {/* banner landmark */}
  <nav aria-label="Primary navigation"></nav>
</header>
<main>  {/* main landmark - most important! */}
  <article>Content</article>
</main>
<aside>  {/* complementary landmark */}
  Related links
</aside>
<footer>  {/* contentinfo landmark */}
  Copyright
</footer>
```

**Q6: How do you make tables accessible?**

**A:** Accessible tables require proper structure and clear relationships between headers and data:

**Key elements:**

1. **`<caption>`**: Provides title/description of table
2. **`<thead>`, `<tbody>`, `<tfoot>`**: Group related rows
3. **`<th>` with scope**: Define column and row headers
4. **scope attribute**: "col" for column headers, "row" for row headers
5. **Complex tables**: Use headers attribute to reference multiple headers

```typescript
<table>
  <caption>Q4 2024 Sales by Region</caption>
  <thead>
    <tr>
      <th scope="col">Region</th>
      <th scope="col">Sales</th>
      <th scope="col">Growth</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">North</th>
      <td>$150K</td>
      <td>+15%</td>
    </tr>
  </tbody>
</table>
```

**For complex tables with multiple header levels:**
```typescript
<table>
  <tr>
    <th id="region" scope="col">Region</th>
    <th id="q1" scope="col">Q1</th>
    <th id="q2" scope="col">Q2</th>
  </tr>
  <tr>
    <th headers="region" scope="row">North</th>
    <td headers="north q1">100</td>
    <td headers="north q2">125</td>
  </tr>
</table>
```

**Don't:**
- Use tables for layout purposes
- Omit `<th>` elements entirely
- Use `<td>` for headers
- Create overly complex tables if simpler alternatives exist

**Q7: What's the first rule of ARIA and what does it mean in practice?**

**A:** The first rule of ARIA is: "If you can use a native HTML element or attribute with the semantics and behavior you require already built in, instead of re-purposing an element and adding an ARIA role, state or property to make it accessible, then do so."

**What this means in practice:**

```typescript
// ❌ BAD: Re-purposing div with ARIA
<div 
  role="button" 
  tabIndex={0}
  onClick={handleClick}
  onKeyDown={(e) => {
    if (e.key === 'Enter' || e.key === ' ') handleClick();
  }}
>
  Click me
</div>

// ✅ GOOD: Native button element
<button onClick={handleClick}>
  Click me
</button>
```

**Why native elements are better:**

1. **Built-in keyboard support**: Native buttons respond to Enter and Space automatically
2. **Screen reader support**: Announced correctly without extra ARIA
3. **Form integration**: Native buttons work with form submission
4. **Browser consistency**: Less cross-browser testing needed
5. **Less code**: Simpler and more maintainable

**When ARIA is appropriate:**
- Custom widgets not available in HTML (tabs, tree views, sliders)
- Dynamic content updates (aria-live regions)
- Providing additional context (aria-label, aria-describedby)
- Complex interactions requiring state management (aria-expanded, aria-selected)

**The hierarchy:**
1. Native HTML semantic elements (best)
2. ARIA roles on semantic elements (when needed)
3. ARIA roles on generic elements (last resort)

Always question whether you really need ARIA before adding it. Often, the answer is no.

**Q8: How does semantic HTML improve SEO?**

**A:** Semantic HTML improves SEO (Search Engine Optimization) by providing clear signals to search engines about content structure and meaning:

**How it helps:**

1. **Content hierarchy**: Search engines use heading levels (h1-h6) to understand document structure and importance
2. **Landmark identification**: Elements like `<article>`, `<nav>`, `<aside>` help search engines identify content types
3. **Content prioritization**: `<main>` signals primary content, `<aside>` signals less important content
4. **Rich snippets**: Semantic elements can trigger special search result displays
5. **Better crawling**: Clear structure helps search bots efficiently parse content
6. **Mobile search**: Semantic landmarks help with mobile-first indexing

**Specific examples:**

```typescript
// ✅ GOOD for SEO
<article>
  <h1>Complete Guide to React Hooks</h1>  {/* Primary keyword in h1 */}
  <p>Published on <time datetime="2024-01-15">January 15, 2024</time></p>
  
  <section>
    <h2>What are React Hooks?</h2>  {/* Section keywords in h2 */}
    <p>React Hooks are...</p>
  </section>
  
  <section>
    <h2>Common Hooks</h2>
    <p>The most frequently used hooks...</p>
  </section>
</article>

// ❌ BAD for SEO
<div class="post">
  <div class="title">Complete Guide to React Hooks</div>  {/* Not a heading */}
  <div class="date">January 15, 2024</div>  {/* Not semantic date */}
  <div class="section">
    <div class="subtitle">What are React Hooks?</div>
    <div>React Hooks are...</div>
  </div>
</div>
```

**Additional semantic elements that help SEO:**
- `<address>`: Contact information
- `<time>`: Dates and times
- `<mark>`: Highlighted or relevant text
- `<strong>` and `<em>`: Emphasis and importance
- `<cite>`: Citations and references
- `<figure>` and `<figcaption>`: Images with captions

Search engines increasingly use semantic signals to understand content, making semantic HTML not just an accessibility best practice but also an SEO best practice.

## Key Takeaways

- Semantic HTML provides implicit accessibility features that ARIA must explicitly define
- The accessibility tree is derived from the DOM and consumed by assistive technologies
- Every semantic element has an implicit ARIA role that shouldn't be overridden
- Proper heading hierarchy (h1-h6) creates document outline for navigation
- Landmark regions (header, nav, main, aside, footer) enable quick navigation
- Lists (ul, ol, dl) provide context about the number and type of items
- Tables need proper structure with caption, thead, th with scope attributes
- Use `<button>` for actions, `<a>` for navigation
- Generic divs and spans should be last resort after semantic options exhausted
- The first rule of ARIA: don't use ARIA if semantic HTML provides what you need
- Semantic HTML benefits accessibility, SEO, maintainability, and code clarity
- Test your accessibility tree in browser DevTools to verify semantic structure
- Invalid or non-semantic HTML creates gaps in the accessibility tree
- Retrofitting accessibility is harder than building it in from the start
