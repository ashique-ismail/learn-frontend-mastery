# Package Managers: npm, pnpm, and Yarn (Lockfiles, Workspaces)

## The Idea

**In plain English:** A package manager is a tool that automatically downloads and organizes ready-made code libraries (called "packages") that your project needs, so you don't have to track down and install each one yourself. Think of a package as a bundle of pre-written code someone else made that you can plug into your own project.

**Real-world analogy:** Imagine you are opening a restaurant and instead of growing every ingredient yourself, you hand a shopping list to a professional grocery service. They go to the store, buy exactly the items on your list (in the exact brands and quantities you specified), and restock your kitchen whenever the list changes.
- The shopping list = your `package.json` (the file that says which packages your project needs)
- The receipt showing exact brand, size, and price of every item bought = the lockfile (`package-lock.json`, `yarn.lock`, or `pnpm-lock.yaml`)
- The grocery service that reads the list and does the fetching = the package manager (npm, Yarn, or pnpm)

---

## Overview

Package managers are the dependency resolution, installation, and workflow tools that underpin every modern JavaScript project. npm ships with Node.js and is ubiquitous; Yarn introduced deterministic installs and workspaces; pnpm re-architected the storage model around a content-addressable store and symlinked node_modules. Understanding the internals — how each constructs `node_modules`, how lockfiles ensure reproducibility, and how workspaces manage monorepos — is critical for senior engineers who own CI pipelines, debug phantom dependency issues, or architect monorepo toolchains.

## npm: The Default

### Core Concepts

npm (Node Package Manager) ships with Node.js and manages packages from the npm registry. Its key files are `package.json` (dependencies and scripts declared by the developer) and `package-lock.json` (exact locked dependency graph generated by npm).

```bash
# Install all dependencies
npm install

# Add a production dependency
npm install express

# Add a dev dependency
npm install --save-dev vitest

# Add an exact version (no semver range)
npm install --save-exact lodash

# Update a package
npm update react

# Remove a package
npm uninstall lodash

# Run a script
npm run build

# Execute a bin without installing globally (npx)
npx create-react-app my-app
```

### package.json: Dependency Ranges

```json
{
  "dependencies": {
    "express": "^4.18.2",
    "lodash": "~4.17.0",
    "uuid": "9.0.0"
  },
  "devDependencies": {
    "vitest": "^1.0.0",
    "typescript": "5.3.3"
  },
  "peerDependencies": {
    "react": ">=18.0.0"
  },
  "optionalDependencies": {
    "fsevents": "^2.3.0"
  }
}
```

Semver range semantics:
- `^4.18.2` — compatible: `>=4.18.2 <5.0.0` (patch and minor updates allowed)
- `~4.17.0` — approximately: `>=4.17.0 <4.18.0` (patch updates only)
- `4.18.2` or `=4.18.2` — exact version
- `*` — any version (dangerous)
- `>=4.18.2 <5.0.0` — explicit range

### npm's node_modules: Flat Tree

npm v3+ uses a flat `node_modules` structure. Dependencies of dependencies are hoisted to the root `node_modules` when there are no version conflicts. When a conflict exists, the conflicting version is nested inside the package that requires it.

```
node_modules/
  express/           <- hoisted
  lodash/            <- hoisted (version 4.x)
  some-package/
    node_modules/
      lodash/        <- nested (requires version 3.x)
```

This hoisting is the source of "phantom dependencies" — your code can `require('express')` even without declaring it in `package.json`, because Express's dependencies are hoisted to root. This is a correctness issue.

### package-lock.json

The lockfile records the exact resolved version and content hash of every package in the tree:

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "lockfileVersion": 3,
  "requires": true,
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-...",
      "dependencies": {
        "accepts": "~1.3.8"
      }
    }
  }
}
```

Key points:
- `npm install` with an existing lockfile installs exactly what the lockfile specifies (not semver ranges)
- `npm ci` strictly installs from lockfile and deletes existing `node_modules` first — use in CI
- `npm install` updates the lockfile if ranges can be resolved to newer versions
- Always commit `package-lock.json` to version control

```bash
# CI-appropriate install — fails if lockfile is out of sync
npm ci

# Developer install — may update lockfile
npm install
```

## Yarn: Deterministic Installs and Workspaces

### Yarn Classic (v1)

Yarn v1 introduced several improvements over npm v3: parallel downloads, offline caching, and a `yarn.lock` format that was more human-readable than npm's.

```bash
yarn install          # install all dependencies
yarn add react        # add production dependency
yarn add -D vitest    # add dev dependency
yarn remove lodash    # remove dependency
yarn upgrade react    # update to latest in range
yarn run build        # run script
yarn dlx create-vite  # equivalent of npx
```

### yarn.lock Format

```yaml
# yarn.lock
express@^4.18.0:
  version "4.18.2"
  resolved "https://registry.yarnpkg.com/express/-/express-4.18.2.tgz#..."
  integrity sha512-...
  dependencies:
    accepts "~1.3.8"
    array-flatten "1.1.1"
```

### Yarn Berry (v2+): Plug'n'Play

Yarn Berry fundamentally changed the node_modules model. Its flagship feature is **Plug'n'Play (PnP)**: instead of unpacking packages into `node_modules`, Yarn maintains a single content-addressed cache and generates a `.pnp.cjs` loader that maps package resolution at require time.

```bash
# Enable Yarn Berry in a project
yarn set version berry

# With PnP, there is no node_modules folder
# Packages are in the global cache, referenced via .pnp.cjs
```

```javascript
// .pnp.cjs maps package specifiers to cache paths
// Node's require() is patched to use this map
// Result: faster installs, no disk duplication, strict isolation
```

**Zero-installs**: With Yarn Berry, you can commit the `.yarn/cache/` folder and avoid running `yarn install` in CI at all:

```gitignore
# .gitignore with zero-installs
.yarn/cache/    <- NOT ignored, committed to repo
.pnp.cjs        <- NOT ignored, committed to repo
node_modules/   <- ignored (doesn't exist with PnP)
```

### Yarn Workspaces

Workspaces enable a monorepo where multiple packages share a single `node_modules` at the root:

```json
// root package.json
{
  "private": true,
  "workspaces": [
    "packages/*",
    "apps/*"
  ]
}
```

```
monorepo/
  package.json        <- root with workspaces config
  yarn.lock           <- single lockfile for all packages
  node_modules/       <- shared (all non-conflicting deps hoisted here)
  packages/
    ui/
      package.json
    utils/
      package.json
  apps/
    web/
      package.json
    api/
      package.json
```

```bash
# Run a command in a specific workspace
yarn workspace @company/ui build

# Add a dependency to a specific workspace
yarn workspace @company/web add react

# Run a script in all workspaces
yarn workspaces run test
```

Cross-workspace dependencies use the `workspace:` protocol to ensure local packages are linked rather than pulled from the registry:

```json
{
  "name": "@company/web",
  "dependencies": {
    "@company/ui": "workspace:*",
    "@company/utils": "workspace:^"
  }
}
```

## pnpm: Content-Addressable Store and Strict Isolation

### The pnpm Storage Model

pnpm is the most architecturally distinct package manager. Every package version is stored once in a global content-addressable store (`~/.pnpm-store`). Project `node_modules` are populated with hard links (or symlinks on Windows) to the store — disk usage is dramatically reduced.

```
~/.pnpm-store/            <- global store
  v3/
    files/
      00/abc123...        <- content-addressed file blobs

project/node_modules/
  .pnpm/                  <- all packages in a flat virtual store
    express@4.18.2/
      node_modules/
        express -> hard link to store
  express -> symlink to .pnpm/express@4.18.2/node_modules/express
```

### Strict Isolation: No Phantom Dependencies

pnpm's key correctness feature: it does not hoist packages to the root `node_modules`. Each package can only access its declared dependencies. If `express` is a dependency of `some-other-package` but not declared in your `package.json`, your code cannot require it — Node.js will follow the symlinks and find no valid resolution.

```javascript
// This would succeed in npm/yarn (phantom dependency)
// This FAILS with pnpm if express is not in your package.json
const express = require('express'); // Error: Cannot find module 'express'
```

This strictness catches undeclared dependencies at development time, preventing "works on my machine" CI failures.

```bash
# pnpm basic commands
pnpm install
pnpm add express
pnpm add -D vitest
pnpm remove lodash
pnpm run build
pnpm dlx create-vite

# Run in specific workspace
pnpm --filter @company/web run build

# Run in all workspaces
pnpm -r run test
```

### pnpm-lock.yaml

```yaml
lockfileVersion: '6.0'

settings:
  autoInstallPeers: true
  excludeLinksFromLockfile: false

dependencies:
  express:
    specifier: ^4.18.2
    version: 4.18.2

packages:
  /express@4.18.2:
    resolution: {integrity: sha512-...}
    dependencies:
      accepts: 1.3.8
    dev: false
```

### pnpm Workspaces

pnpm uses a `pnpm-workspace.yaml` file:

```yaml
# pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'apps/*'
  - '!**/test/**'
```

```bash
# Install all workspace dependencies
pnpm install

# Add to specific package
pnpm add react --filter @company/web

# Run script in all packages
pnpm -r run build

# Run only in changed packages (requires changesets or turbo)
pnpm --filter ...[origin/main] run build
```

| Registry support | npm + custom | npm + custom | npm + custom | npm + custom |

## Lockfile Deep Dive

### Why Lockfiles Are Non-Negotiable

Without a lockfile, `npm install` resolves semver ranges at install time. `react@^18.0.0` may resolve to `18.2.0` in January and `18.3.0` in March. If `18.3.0` has a breaking change or bug, your application breaks in March without any code change.

With a lockfile:
- `npm ci` reproduces the exact tree from the lockfile, ignoring semver ranges
- Every developer, every CI job, every deployment gets identical packages
- Security audits and vulnerability fixes are trackable via lockfile diffs

### Lockfile Merge Conflicts

Lockfiles frequently cause merge conflicts in active monorepos. The correct resolution:

```bash
# After resolving package.json conflicts, regenerate the lockfile
npm install       # regenerates package-lock.json
yarn install      # regenerates yarn.lock
pnpm install      # regenerates pnpm-lock.yaml

# Then commit the regenerated lockfile
```

Never manually edit a lockfile. Accept one side of the conflict and regenerate.

### Auditing and Security

```bash
# Check for known vulnerabilities
npm audit
npm audit fix           # auto-fix non-breaking
npm audit fix --force   # fix including breaking changes (use carefully)

yarn audit
pnpm audit
```

## Best Practices

### 1. Always Commit Lockfiles

Lockfiles are part of the reproducibility contract. Never `.gitignore` them.

### 2. Use npm ci in CI/CD

```yaml
# GitHub Actions
- name: Install dependencies
  run: npm ci
```

`npm ci` is faster than `npm install` in CI (skips lockfile updates, deletes old node_modules), and fails if `package-lock.json` is out of sync with `package.json`.

### 3. Use Exact Versions for Internal Tools

```bash
npm install --save-exact typescript@5.3.3
# Results in "typescript": "5.3.3" — no range
# Prevents unexpected TypeScript version changes from breaking builds
```

### 4. Set engines Field

```json
{
  "engines": {
    "node": ">=20.0.0",
    "npm": ">=10.0.0"
  }
}
```

Prevents installation with incompatible Node.js versions.

### 5. Use .npmrc for Registry Configuration

```ini
# .npmrc
registry=https://registry.npmjs.org/
@company:registry=https://npm.company.com/
engine-strict=true
save-exact=true
```

## Interview Questions

### Q1: What is the difference between npm install and npm ci?

**Answer:** `npm install` reads `package.json`, resolves semver ranges, and updates `package-lock.json` if newer compatible versions are available. It is intended for development use when you want to update dependencies. `npm ci` reads `package-lock.json` exclusively, installs exactly those versions, and fails if `package-lock.json` is not in sync with `package.json`. It also deletes the existing `node_modules` before installing, ensuring a clean state. `npm ci` is the correct command for CI/CD pipelines because it guarantees reproducibility and is faster.

### Q2: What is a phantom dependency and why does pnpm prevent it?

**Answer:** A phantom dependency is a package your code `require()`s that is not declared in your `package.json` but happens to be available because npm/yarn hoisted it to the root `node_modules` as a transitive dependency of something else. For example, if you depend on `express` and express depends on `qs`, npm hoists `qs` to the root — your code can require it without declaring it. This is brittle: if you stop using express, or if express drops `qs` as a dependency, your code breaks. pnpm prevents this by not hoisting packages. Each package's `node_modules` only contains what it explicitly declared, so Node.js module resolution fails at development time for undeclared imports.

### Q3: How does the workspace:* protocol work in a monorepo?

**Answer:** The `workspace:` protocol tells the package manager to resolve the dependency to the local workspace package rather than fetching from the registry. `workspace:*` means "use exactly the current local version." `workspace:^` means "use the current local version with a caret range when publishing." This ensures that during development, `@company/web` always uses the local source of `@company/ui` rather than a potentially stale published version. When you publish a package, the package manager replaces `workspace:*` with the actual current version number in the published tarball.

### Q4: What is Yarn Berry's Plug'n'Play and what problem does it solve?

**Answer:** PnP eliminates `node_modules` directories entirely. Instead, Yarn maintains packages in a global content-addressed cache and generates a `.pnp.cjs` resolution map. When Node.js requires a package, the patched resolver uses the map to find the package in the cache. This solves three problems: disk space waste (no per-project copies), slow `node_modules` generation (millions of file operations), and phantom dependencies (every import is validated against the PnP map). The trade-off is compatibility — some tools hardcode assumptions about `node_modules` existing and require special configuration or SDKs.

### Q5: Why would you choose pnpm over npm or Yarn in a monorepo?

**Answer:** pnpm excels in monorepos for several reasons. Its content-addressable store means that if five packages all depend on `react@18.2.0`, there is exactly one copy of that package on disk, hard-linked to each. In large monorepos with 50+ packages and shared deps, this can reduce `node_modules` size by 50-80%. Strict isolation prevents phantom deps, which become particularly dangerous in monorepos where transitive hoisting is widespread. pnpm's filter syntax (`--filter`) enables fine-grained command targeting. It integrates natively with Turborepo's remote cache. Install times are competitive with Yarn v1 and much faster than npm for large dependency trees.

## Common Pitfalls

### 1. Mixing Package Managers in the Same Repo

```json
// Prevent this with packageManager field
{
  "packageManager": "pnpm@8.14.0"
}
```

Using `npm install` in a pnpm project creates a `package-lock.json` alongside `pnpm-lock.yaml`, causing inconsistencies.

### 2. Forgetting --legacy-peer-deps After npm v7

npm v7+ enforces peer dependencies strictly. Many packages have incorrect peer dep declarations:

```bash
# Quick workaround (understand the implications)
npm install --legacy-peer-deps

# Better: fix the peer dep in the package or use overrides
```

```json
// package.json overrides — force a specific resolution
{
  "overrides": {
    "some-package>react": "18"
  }
}
```

### 3. Not Pinning Tool Versions

```bash
# Danger: team members get different tool versions
npm install typescript  # may install 5.3 for one, 5.4 for another

# Safe: exact pin
npm install --save-exact typescript@5.3.3
```

### 4. Publishing node_modules

Add `node_modules` to `.gitignore` and `.npmignore` or use the `files` field in `package.json` to whitelist what gets published:

```json
{
  "files": ["dist", "src", "README.md"]
}
```
